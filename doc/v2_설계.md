# HR Data Engine v2 설계

---

## 1. 전체 구조 (현재 상태)

### 파이프라인 흐름

```
RAW (엑셀/CSV)
   ↓
INGEST
   ↓
NORMALIZE
   ↓
CONTACT PARSE (연락망)
   ↓
MERGE
   ↓
ENTITY RESOLUTION
   ↓
ORG SNAPSHOT
   ↓
METRICS
   ↓
STREAMLIT DASHBOARD
```

### 코드 기준 매핑

* ingest → `read_all_excels` (ingest_v2.py)
* normalize → `normalize_v2` → (향후 `normalize_records`)
* contact parsing → `parse_contact_org_xlsx` (contact_org_parser.py)
* pipeline → `run_pipeline` (pipeline.py)
* entity → `resolve_entities` (entity_resolver.py)
* snapshot → `build_org_snapshot` (org_snapshot_builder.py)
* metrics → `build_metrics` (metrics.py)
* app → `app.py (Streamlit)`

---

## 2. 현재 구조의 핵심 문제

### ❌ 문제 1: normalize 구조 한계

* 컬럼 매핑이 단순 (name/org/dept 중심)
* 다양한 HR 양식 대응 불가
* schema_registry와 연결 안됨

👉 개선:

* `normalize_records` 기반 통합 필요

---

### ❌ 문제 2: contact parser 하드코딩

```python
parse_contact_org_xlsx("특정파일.xlsx")
```

문제:

* 특정 회사 의존
* 자동 ingest 불가
* 재사용성 없음

---

### ❌ 문제 3: entity resolution 단순

현재 로직:

1. emp_id
2. name + join_date
3. else new

문제:

* 이름 중복 대응 불가
* 데이터 오류 시 매칭 실패
* 실제 HR 데이터에서 신뢰도 낮음

---

### ❌ 문제 4: snapshot 구조 문제

현재:

* raw 파일 기반 snapshot 생성

문제:

* 파일 구조 의존
* 시점 개념 부족
* history 기반 아님

👉 개선 방향:

* employment_history 기반 snapshot 생성

---

### ❌ 문제 5: schema_registry 미연결

현재:

* schema_registry 별도 실행
* pipeline과 분리됨

문제:

* 자동화 불가능
* 데이터 분류 의미 없음

---

## 5. 핵심 개선 설계

---

### [A] ingest 구조 개선

기존:

```
read_all_excels → 전체 읽기
```

개선:

```
1. 파일 스캔
2. schema_registry 기반 유형 판별
3. 유형별 처리 (plugin 구조)
4. bronze 저장
```

---

### [B] normalize 통합

기존:

* normalize_v2
* normalize_records (분리됨)

개선:

```
normalize_records = 단일 표준화 엔진
```

기능:

* 컬럼 표준화
* 날짜 변환
* 조직 path 생성
* skill 정규화
* review flag 생성

---

### [C] contact parser 개선

기존:

```
pipeline에서 직접 호출
```

개선:

```
if file_kind == contact:
    contact parser 실행
```

→ ingest 단계에서 자동 처리

---

### [D] entity resolution 고도화

단계별 개선:

1단계:

* emp_id
* name + join_date

2단계:

* name + org + title fuzzy

3단계:

* phone / email 매칭

4단계 (확장):

* identity graph 기반 매칭

---

### [E] snapshot 구조 재설계

기존:

* raw 기반 snapshot

개선:

```
employment_history → 기준일 snapshot 생성
```

※ 기존 snapshot_builder는 deprecated 예정

---

### [F] metrics 확장

기존:

* headcount

개선:

* turnover rate
* early exit
* 조직별 성장률
* 이동 분석
